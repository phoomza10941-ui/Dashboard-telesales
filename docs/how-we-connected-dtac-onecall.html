How We Connected dtac OneCall → the Dashboard

Pulling each agent's Talk Time into the Supervisor page — explained simply

Topic: lib/oreka.ts · Talk Time integration
TL;DR (the one-sentence version)
dtac OneCall's call-recording website is secretly running software called Oreka, which has a hidden REST API. Our server logs into that API, downloads the list of today's calls, adds up the seconds per phone number, and shows it as Talk Time per agent on /supervisor. No CSV exports, no manual work — it updates live.

🧠 The plain-English analogy

Think of dtac OneCall like a library of phone calls:

  • The website you normally click around in = the front desk.
  • The API we found = a staff-only back door that hands you the raw catalog.
  • Our server walks in the back door with a username + password, gets a visitor badge (token), then asks: "give me every call from today."
  • We match each call's phone number to an agent, and sum the seconds talked. That total is the Talk Time.

1 · What we discovered

The key breakthrough that made all of this possible:

The voice-record portal at onecallvoicerecord.dtac.co.th/orkui/ is not a custom dtac system — it's an off-the-shelf product called Oreka (OrecX). Oreka ships with a documented REST API called OrkTrack, sitting at:

https://onecallvoicerecord.dtac.co.th/orktrack/rest/

That means instead of a human logging in and exporting spreadsheets, our server can fetch the data directly and live. This is the whole reason the integration is clean.

2 · How the connection works (the flow)

Everything runs on the server only (never in the user's browser), so the password is never exposed. Here is the journey of one page load:

Supervisor opens /supervisor/talk-time │ ▼ ┌───────────────────────┐ 1. LOGIN (username + password) │ Our Next.js server │ ───────────────────────────────────▶ Oreka OrkTrack API │ (lib/oreka.ts) │ ◀─────────────────────────────────── returns "accesstoken" └───────────────────────┘ │ 2. GET /recordings (with token, page by page) │ ────────────────────────────────────────────▶ Oreka │ ◀──────────────────────────────────────────── list of today's calls │ │ 3. Add up duration (seconds) grouped by phone number │ 4. Match phone number → agent nickname (Supabase profiles) ▼ ┌───────────────────────┐ │ Talk Time per agent │ → shown on the Supervisor page └───────────────────────┘
  1. Log in. The server sends a POST to /orktrack/rest/user/login with an Authorization: Basic base64(user:password) header. Oreka replies with a token in the JSON field accesstoken.
  2. Ask for the calls. Using that token, the server calls GET /orktrack/rest/recordings for today's date range. Big days have thousands of calls, so we read them in pages of 1000, five pages at a time (in parallel) for speed.
  3. Add up the seconds. Each call record has a duration (in seconds) and a localParty (the agent's phone number). We group by phone number and sum the durations → that's Talk Time.
  4. Put a name to the number. We look up the phone number in our Supabase profiles table (column oreka_ext) to turn +66… into a human nickname.
  5. Cache & show. The result is cached for ~90 seconds (so refreshing the page doesn't hammer Oreka) and rendered on /supervisor.

3 · The important details (so it's correct)

📞 Matching a call to an agent

Each agent has a fixed phone number. We store it on their profile and join on it — numbers are unique and stable, far more reliable than matching by name.

Oreka fieldMeaningMatches to
localPartyThe agent's number (+66…)profiles.oreka_ext → nickname
remotePartyThe customer's number—
durationTalk time in secondssummed per agent
directionIN / OUTcounted as in/out calls
⚠️ Get this right: localParty is the full number in +66… form — not a short extension. Numbers are auto-normalized (0… / 66… → +66…) when a supervisor enters them in Supervisor → Settings → "Talk Time — จับคู่เบอร์ Oreka".

🕐 Timezone (a classic gotcha)

Oreka stores every timestamp and startdate in UTC. Thailand is UTC+7. So to ask for "today in Thailand", we shift the Thai day boundaries back 7 hours before sending the query, then display the totals back in Thai time. If we skipped this, the numbers would be off by up to a day.

👥 Two call-center accounts

There are two Oreka logins — Gosell and Hopeful — same API, different credentials. The server fetches both and merges the results. If one account's credentials are missing or it fails, the other still works.

🛟 It never breaks the page

If Oreka is down, slow, or times out, the function getTalkTimeByAgentSafe() catches the error and returns empty data with a message — so /supervisor still loads fine, it just shows "talk time unavailable" instead of crashing.

4 · The login call, concretely

POST /orktrack/rest/user/login?version=orktrack&accesspolicy=all&licenseinfo=true
Header:  Authorization: Basic base64("+66xxxxxxxxx:password")
Body:    (empty)

→ Response JSON:
   { "accesstoken": "a1b2c3d4-....", ... }     ← this token is reused for every request

The token is cached in memory. If a later request returns 401 or 403 (token expired), the server automatically logs in again and retries — no human involved.

7 · Common questions

Is the dtac password safe?
Yes. Credentials live only in .env.local (never committed to git), and Oreka is only ever called from the server. The browser never sees the password or the token.
Why do we cache for ~90 seconds?
Supervisors refresh the page often. Without a cache, every refresh would re-download thousands of call records and hammer the dtac API. 90 seconds keeps the data feeling "live" while staying gentle on Oreka. Past days are cached longer (5–10 minutes) because they no longer change.
What if a new agent's talk time shows as a bare number?
That agent's phone number isn't mapped yet. Go to Supervisor → Settings → "Talk Time — จับคู่เบอร์ Oreka" and enter their number. It auto-converts to +66… and the next refresh shows their nickname.
Does this touch War Room or My-Desk?
No — by design, Talk Time appears only on the Supervisor page.
How do I debug it if numbers look wrong?
Run node scripts/test-oreka.mjs — it reads your .env.local and hits the Oreka API directly, so you can see the raw records without going through the dashboard.
