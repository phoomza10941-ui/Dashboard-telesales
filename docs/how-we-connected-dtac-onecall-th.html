01
🔑 ตั้งค่า Credential ใน .env Configuration

เก็บ Username และ Password ของบัญชี dtac OneCall ไว้ใน Environment Variable เพื่อให้ server อ่านได้ฝั่งเดียว — ไม่โผล่ฝั่ง browser เด็ดขาด

.env.local
OREKA_BASE_URL=https://onecallvoicerecord.dtac.co.th
OREKA_USER=+66xxxxxxxxx # เบอร์มือถือที่ใช้ login
OREKA_PASSWORD=••••••••

# ถ้ามีหลายทีม ก็เพิ่ม account ที่ 2 ได้
OREKA_HOPEFUL_USER=+66xxxxxxxxx
OREKA_HOPEFUL_PASSWORD=••••••••
✓ ไม่ commit ขึ้น git server-side only รองรับหลาย account
02
🔐 Server Login ขอ Token Authentication

ตอนมีคนเปิดหน้า /supervisor/talk-time, server จะ POST ไปหา Oreka API โดยใส่ Basic Auth ใน Header → Oreka ตอบกลับด้วย accesstoken (GUID) Token นี้ถูก cache ไว้ใน memory — ไม่ต้อง login ซ้ำทุก request

HTTP Request → Oreka
POST /orktrack/rest/user/login
?version=orktrack&accesspolicy=all

Authorization: Basic base64(user:password)

─── Response ────────────────────────────
{
  "accesstoken": "a1b2c3d4-xxxx-xxxx-xxxx-...",
  "scope": "VIEW_RECORDINGS", ...
}
cache ใน memory re-login อัตโนมัติถ้า 401/403
03
📞 ดึง Recording ของวันนี้ Fetch Recordings

เอา Token ไปยิง GET recordings โดยกำหนดช่วงเวลา "วันนี้ตามเวลาไทย" (UTC+7 → แปลงเป็น UTC ก่อนส่ง) Oreka จะส่ง list สายทั้งหมดกลับมา — แต่ละสายมี เบอร์คนโทร, ระยะเวลา, ทิศทาง ถ้ามีหลายหน้าก็ paginate จนหมด

HTTP Request → Oreka
GET /orktrack/rest/recordings
?range=custom
&startdate=20260615_170000 # 00:00 ไทย = 17:00 UTC วันก่อน
&enddate=20260616_165959 # 23:59 ไทย = 16:59 UTC
&pagesize=1000

Authorization: a1b2c3d4-xxxx-xxxx-xxxx-...

─── Response ────────────────────────────
{
  "objects": [
    { "id": 100659220,
      "localParty": "+66812345678", ← เบอร์ agent
      "duration": 183, ← วินาที
      "direction": "OUT",
      "timestamp": "2026-06-15 10:23:27" }, ← UTC
    ...
  ],
  "nextPageUri": "...&page=2" ← มีหน้าต่อ
}
1,000 records/page fetch 5 pages พร้อมกัน cache 90 วินาที
04
🔗 จับคู่เบอร์โทรกับชื่อ Agent Match Agents

ใน Supabase มีคอลัมน์ profiles.oreka_ext เก็บเบอร์ dtac ของแต่ละ agent ไว้ (รูปแบบ +66...) เราเอา localParty จาก Oreka มา join กับค่านี้ → รู้ว่าสายนี้เป็นของ "เก้า", "ออย", "มิ้น" ฯลฯ

Supabase · profiles table
┌──────────────┬──────────────┬──────────────────┐
│ nickname │ agent_code │ oreka_ext │
├──────────────┼──────────────┼──────────────────┤
│ เก้า │ A001 │ +66812345678 │
│ ออย │ A002 │ +66823456789 │
│ มิ้น │ A003 │ +66834567890 │
└──────────────┴──────────────┴──────────────────┘

# Oreka.localParty "+66812345678" → nickname "เก้า"
# รวม duration ทุกสายของ "+66812345678" = total talk time ของ "เก้า"
ตั้งค่าได้ใน Supervisor → Settings auto-normalize 0xx → +66xx
05
📊 แสดงผลใน Dashboard Display & Fallback

รวม duration ทุกสาย group by ชื่อ agent → ได้ยอดรวม Talk Time (วินาที → HH:MM:SS) แสดงใน /supervisor/talk-time ถ้า Oreka ล่มหรือ timeout ระบบแสดงข้อความแจ้งเตือนแทน — หน้า Supervisor ยังโหลดได้ปกติ

ผลลัพธ์ที่ได้ · AgentTalkTime[]
[
  { nickname: "เก้า",
    totalSeconds: 12480, ← 3 ชม. 28 นาที
    callCount: 47,
    outCount: 39, inCount: 8 },
  { nickname: "ออย",
    totalSeconds: 9840, ... },
  ...
]

# ถ้า Oreka ล่ม → graceful fallback
{ data: [], error: "Oreka ไม่พร้อมใช้งาน" }
✓ graceful fallback เรียงตาม talk time มากสุด cache 90s

// สรุปภาพรวม Flow

ใครเรียก API Server เท่านั้น (Next.js Server Component) — browser ไม่รู้จัก Oreka เลย
Token จัดการ Cache ใน server memory, auto re-login เมื่อหมดอายุ (401/403)
Timezone Oreka เก็บเป็น UTC → แปลงขอบวันไทย (UTC+7) ก่อนส่ง query
จับคู่ Agent Oreka.localParty → profiles.oreka_ext → nickname
Performance Cache 90 วินาที + ดึง 5 หน้าพร้อมกัน (parallel fetch)
ถ้า Oreka ล่ม หน้า /supervisor ยังโหลดได้ แสดงข้อความแจ้งแทน (graceful fallback)
